HashMap源码分析

HashMap 简介

HashMap 主要用来存放键值对，它基于哈希表的 Map 接口实现，是常用的 Java 集合之一，非线程安全的。

HashMap 可以存储 null 的 key 和 value，但 null 作为键只能有一个，null 作为值可以有多个

JDK1.8 之前 HashMap 由 数组+链表 组成的，数组是 HashMap 的主体，链表则是主要为了解决哈希冲突而存在的（“拉链法”解决冲突）。

1.8之后

结构变为数组+链表+（红黑树）

JDK1.8 以后的 HashMap 在解决哈希冲突时有了较大的变化，当链表长度大于阈（yu）值（默认为 8）（将链表转换成红黑树前会判断，如果当前数组的长度小于 64，那么会选择先进行数组扩容（扩容方法中有原因）），而不是转换为红黑树）时，将链表转化为红黑树，以减少搜索时间。

HashMap 默认的初始化大小为 16。之后每次扩充，容量变为原来的 2 倍。并且， HashMap 总是使用 2 的幂作为哈希表的大小。

用二进制表示为00000000 00000000 00000000 00010000，当扩容时,会将容量左移一位即00010000,右数第5位为1,容量变为32，结论：hashmap的容量用二进制表示时,始终只有一位为1,其余的全为0.

底层数据结构分析

#JDK1.8 之前

Hashmap 通过 key 的 hashcode，经过扰动函数

（扰动函数指的就是 HashMap 的 hash 方法，防止一些实现比较差的 hashcode 方法产生碰撞，即目的是——>减少碰撞）

处理，——>得到 hash 值，然后将（n-1）&hash 值判断元素在数组中的位置（数组中不放值）

&：按位与运算

碰撞过程 ——》存在元素 ——》碰撞——》元素 hash 是否相同，key是否相同——》相同直接覆盖

​ 不相同——》拉链法，即将冲突的值加到链表中

JDK 1.8 的 hash 方法 相比于 JDK 1.7 hash 方法更加简化，但是原理不变。

JDK1.8 之后

JDK1.8 以后在解决哈希冲突时有了较大的变化。

treeifyBin（）方法——》把元素变为树结构

注：当一个数组位置上集中了多个键值对，那是因为这些key的hash值和数组长度取模之后结果相同。（并不是因为这些key的hash值相同）

HashMap类的属性：

  • loadFactor 加载因子（0<loadFactor <1）

    loadFactor 加载因子是控制数组存放数据的疏密程度，数据越多，越趋近于1

    太大——》查找效率低

    太小——》利用率低，数据很分散

    0.75f 是官方给出的一个比较好的临界值。

  • threshold默认是12 美[ˈθreʃhoʊld]

    threshold = capacity * loadFactor，当 Size>=threshold，对数组扩增，这个的意思就是 衡量数组是否需要扩增的一个标准。

    所以数量达到 16*0.75=12 ——》扩容——》rehash复制数据——》消耗性能

#构造方法

HashMap 中有四个构造方法，它们分别如下：

putMapEntries 方法：用传入的map做内容初始化

put 方法

HashMap 只提供了 put 用于添加元素，putVal 方法只是给 put 方法调用的一个方法，并没有提供给用户使用。

对 putVal 方法添加元素的分析如下：

  1. 如果定位到的数组位置没有元素 就直接插入。
  2. 如果定位到的数组位置有元素就和要插入的 key 比较，如果 key 相同就直接覆盖，如果 key 不相同，就判断 p 是否是一个树节点，如果是就调用e = ((TreeNode<K,V>)p).putTreeVal(this, tab, hash, key, value)将元素添加进入。如果不是就遍历链表插入(插入的是链表尾部)。

流程图

说明:上图有两个小问题：

  • 直接覆盖之后应该就会 return，不会有后续操作。参考 JDK8 HashMap.java 658 行（issue#608 (opens new window)）。
  • 当链表长度大于阈值（默认为 8）并且 HashMap 数组长度超过 64 的时候才会执行链表转红黑树的操作，否则就只是对数组扩容。参考 HashMap 的 treeifyBin() 方法（issue#1087 (opens new window)）。

总结一下：

  1. 即put办法，先判断桶是否为空，如果为空，就进行初始化，如果不为空，就计算key的hashcode经过扰动函数，即hash函数的值并与n-1做按位与运算计算值，求得数组下标，

  2. 如果桶的该位置为空，就直接放入，如果有值，就比较hashcode和key的值，如果相等，则直接覆盖，如果不等，就用拉链法处理，

  3. 先判断该节点是否为树的节点，如果是，就变为树的的节点，放在红黑树上，如果不为树的节点，就只有链表，尾插法，判断链表数量是否大于8，如果成立，就开始进行树化，在树化函数内先判断桶的数量是否大于64，如果大于，进行树化，如果不等，就对桶进行扩容，

  4. 最后将容量加1，判断是否大于大于临界值，如果大于，就进行扩容。

我们再来对比一下 JDK1.7 put 方法的代码

对于 put 方法的分析如下：

  • ① 如果定位到的数组位置没有元素 就直接插入。
  • ② 如果定位到的数组位置有元素，遍历以这个元素为头结点的链表，依次和插入的 key 比较，如果 key 相同就直接覆盖，不同就采用头插法插入元素。

resize 方法

进行扩容，会伴随着一次重新 hash 分配，并且会遍历 hash 表中所有的元素，是非常耗时的。在编写程序中，要尽量避免 resize。

总结以下，resize即，

  1. 先判断是否大于能承受的最大值，即2的30次方，如果大于，就直接将数组设为最大值，

  2. 如果小于，就直接将容量变为原来的2倍，即oldcap= oldcap向左移一位，并计算新的临界值，

  3. 最后将原map中的数据重新分配，这个过程十分消耗资源，所以不要经常对HashMap扩容。
